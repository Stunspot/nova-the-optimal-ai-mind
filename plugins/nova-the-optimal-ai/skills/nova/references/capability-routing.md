# Nova capability routing

Use this map only when direct work is insufficient. The trigger is a required transformation, not a matching noun.

| Needed transformation | Route | Return to Nova | Do not activate merely because |
|---|---|---|---|
| Time-bounded current research across sources | `$omnara-deep-research` | sourced synthesis, confidence, gaps | the user asks one stable factual question |
| Canon, provenance, version conflict, knowledge custody | `$rupert-giles-knowledge-steward` | canonical object or custody decision | a document simply needs summarizing |
| Corpus-grounded answer or retrieval design | `$retrieval-intelligence` | evidence-bearing retrieval result | web research alone is sufficient |
| Adversarial challenge of retrieval/citation evidence | `$retrieval-reviewer` | accepted limits or defects | first-pass evidence does not yet exist |
| Visual story from evidence | `$signal-loom` | inspectable visual artifact and source boundary | decoration would add no understanding |
| Explicit low-pressure reminder pin | `$corkboard` | observed pin/recall result | the conversation contains a useful detail |
| Governed context about a person or relationship | `$dunbar` | scoped record or recalled context | a name appears in passing |
| Live repository/tool-state orientation and bounded change | `$agentic-coding` | technical work record and observed oracle | the request is merely to discuss software |
| First-pass software/package/release evidence | `$software-verification` | verification package and bounded claims | confidence or a lint pass feels reassuring |
| Independent challenge of existing verification | `$verification-reviewer` | adversarial verdict and residual risk | no reviewable evidence exists yet |
| Game, character, backstory, world, scene, or fiction continuity | `$ludis-continuum` | playable state or compelling creative artifact | ordinary business writing needs flair |
| Two or more distinct MIND Faculty transformations that must reunify | `$augment-of-mind` | one integrated consequential outcome | only one Faculty or direct answer is needed |

## Collision order

1. Honor an explicit named skill when it fits the request and authority.
2. Keep ordinary reversible work direct.
3. Let domain transformation choose the specialist.
4. Let MIND integrate distinct cognitive transformations, not occupational tools.
5. Use TestForge after making software/package change evidence, not as a substitute for the maker.
6. Use reviewers only after an operator has produced inspectable evidence.
7. Preserve Nova as the user-facing owner of the final result.

When several routes are useful, load the smallest sufficient set and sequence dependencies. Do not load the whole estate "just in case." A capability name in the answer is not evidence that its skill was discoverable, read, or active.
