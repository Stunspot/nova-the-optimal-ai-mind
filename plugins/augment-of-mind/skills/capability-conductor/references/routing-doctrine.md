# Routing doctrine

## Route by owned transformation

Activation keywords are clues, not custody. Prefer the capability whose responsibility naturally ends in the required state. When several candidates appear plausible, compare:

1. transformation owned;
2. evidence and artifacts required;
3. authority and safety boundary;
4. host availability;
5. handoff and merge cost;
6. independent-review need.

Do not choose a large ensemble merely because many skills are available. Do not choose a single generalist merely because coordination is inconvenient.

## Mission decomposition

Decompose only at real responsibility boundaries. A useful workstream has a deliverable that can be judged, a bounded dependency surface, and a clear return condition. If two workstreams constantly exchange unsettled assumptions, keep them under one owner until the interface stabilizes.

Give one capability primary custody for each workstream. Supporting capabilities advise or produce named sub-artifacts; they do not silently compete for the same decision.

## Handoff packet

Every handoff answers:

- What part of the user's desired state is yours?
- What state and evidence are authoritative on entry?
- What may you decide or change?
- What must you return?
- How will the return be judged?
- What condition requires escalation or return without completion?

Transmit only relevant context, but never omit a constraint because another capability is expected to remember it.

## Synthesis

The merge owner checks compatibility before summarizing. Reconcile terms, scopes, timeframes, versions, assumptions, and authority. Preserve genuine conflict rather than averaging it into prose. Attach uncertainty to the affected claim or decision.

Integration is complete only when the combined artifact or state satisfies the mission acceptance target—not when every participant has replied.

## Recovery

Classify failure before rerouting:

- **Performance defect:** the capability had adequate affordances and failed.
- **Prompt or package defect:** the capability was not given the needed doctrine, resources, or state surface.
- **Runtime defect:** a required tool, permission, host feature, or dependency was unavailable.
- **Handoff defect:** critical state, authority, or acceptance information was lost.
- **Mission defect:** the desired state or constraints are internally incompatible or materially underspecified.
- **Evaluator defect:** the gate or evidence collection is invalid.

Repair the smallest upstream cause. Preserve the last known-good state. After repeated unchanged failure, change the route rather than repeating the same ceremony.
